# Part XXIX — Graphics Algorithms

Parts I–XXVIII establish the Linux graphics stack from kernel DRM internals through Wayland compositor architecture, Vulkan API layers, browser rendering engines, terminal emulators, multimedia frameworks, and display hardware. Part XXIX steps back from the stack itself to examine what runs *on* it: the algorithms and mathematical techniques that GPU programs execute once the pipeline is correctly constructed. The two chapters here are reference catalogs, not tutorials — they collect in one place the named techniques that graphics, geometry, and compute programmers reach for repeatedly, with enough description to select the right approach and enough references to implement it correctly.

These chapters address three audiences simultaneously. **Graphics application developers** selecting a shading or geometry technique before committing to an implementation will find the trade-off analysis most useful. **Systems and driver developers** performance-budgeting a GPU workload need to know what class of algorithm is running — register pressure, memory access pattern, divergence profile — and the catalog entries surface those characteristics directly. **Browser and web platform engineers** mapping WebGPU or WebGL workloads onto Linux GPU capabilities will find the per-algorithm extension and API-support notes relevant.

The unifying theme is **GPU-resident data parallelism**: every algorithm here is either already data-parallel or is expressed in a parallel form that maps onto Vulkan compute dispatches, vertex/fragment shader invocations, or hardware fixed-function stages. Where a CPU-centric library remains the practical production path (OpenSubdiv's topology refiner, meshoptimizer's encoder, OpenCASCADE's B-rep mesher), the chapter explains the GPU upload and integration pattern so the CPU pre-pass feeds a GPU-resident pipeline.

## Chapters in This Part

**Chapter 204 — Shader Algorithm Catalog: Recipes, References, and Use Cases** is a no-code reference organising 108 named shader techniques across 13 categories: rendering architecture and pipeline, lighting models and PBR, shadow techniques, texture and sampling algorithms, post-processing and screen-space effects, global illumination, ray tracing and path tracing, compute and data-parallel algorithms, GPU-accelerated simulation, neural rendering, hardware-specific and specialised techniques, animation and deformation, and 2D and vector graphics. Each entry follows the same structure: what the algorithm computes, when to use it, named variants worth knowing, costs and limitations, and a primary reference. The catalog is intended as a decision tool at the whiteboard stage — before a line of shader code is written — and as a vocabulary reference for reading GPU-intensive codebases and papers.

**Chapter 208 — GPU Geometry Algorithms: Subdivision, Implicit Surfaces, Skinning, and IK** organises 107 geometry algorithms across 13 categories: surface representation and modeling (subdivision surfaces, NURBS, metaballs, spline tessellation, vector graphics on GPU, CSG, Poisson reconstruction), mesh processing and topology (simplification, remeshing, UV packing, welding, decimation, half-edge structures), spatial data structures (BVH, octrees, k-d trees, hash grids, PVS), differential geometry (curvature, geodesics, parameterisation, medial axis), animation (skeletal skinning, blend shapes, cloth, IK, crowd simulation), physics (rigid body broadphase, softbody FEM, fluid SPH and FLIP, constraint solving), signed-distance fields and volumetric rendering (SDF operations, sphere tracing, volume rendering, opacity micro-maps), terrain and procedural generation (heightmaps, voxel terrain, fractal noise, L-systems), ray tracing geometry (TLAS/BLAS construction, BVH traversal, ray-triangle intersection, deformable BVH), point cloud processing (splatting, Gaussian splatting, surface reconstruction, compression), neural geometry (NeRF, mesh-conditioned diffusion, learned LOD, neural SDF), specialised techniques (hair rendering, water surfaces, decal projection, fur), and GPU primitives (prefix scan, stream compaction, bitonic sort, histogram). The chapter includes an **Algorithm Selection Guide** structured as decision tables by problem type and runtime constraint, covering all 13 categories, to accelerate algorithm selection before implementation.

## How the Chapters Interrelate

Chapter 204 and Chapter 208 are complementary references operating at different granularities of the geometry-and-shading pipeline. Chapter 204 covers the **shading stage**: once geometry is in clip space, what does the fragment or compute shader do to produce the final pixel value? Chapter 208 covers the **geometry stage**: how is geometry constructed, deformed, accelerated for intersection, or procedurally generated before rasterisation or ray-test? In practice, most GPU workloads combine both: a character is skinned (Chapter 208 §39–47), its meshlets are culled and dispatched (Chapter 204 category I), its surface is shaded with PBR (Chapter 204 category II), and ambient occlusion is computed from an acceleration structure (Chapter 208 category IX, Chapter 204 category VI).

Cross-references to the rest of the book are dense. Chapter 204 references Chapter 135 (ray tracing Vulkan API) for ray tracing algorithms, Chapter 154 (GPU-driven rendering) for indirect draw and meshlet patterns, and Chapter 127 (mesh shaders and VRS) for variable-rate shading. Chapter 208 references Chapter 135 for BVH acceleration structure construction, Chapter 127 for mesh shader geometry pipelines, and Chapter 133 (Vulkan compute queues) for async compute geometry passes. Both chapters presuppose the Vulkan compute foundation established in Chapter 25 and the shader compilation pipeline covered in Chapter 152.
